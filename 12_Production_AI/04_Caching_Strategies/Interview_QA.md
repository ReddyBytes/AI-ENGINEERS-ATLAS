# Interview QA — Caching Strategies

## Beginner

**Q1: What is semantic caching and how is it different from exact-match caching?**

<details>
<summary>💡 Show Answer</summary>

**A:** **Exact-match caching** stores the exact input-output pair and only returns a cached result if the new input is byte-for-byte identical. If a user asks "How do I reset my password?", the cache key might be a hash of that string. If they ask "How can I change my password?", it's a different hash — cache miss, even though the answer is the same.

**Semantic caching** uses embeddings to find "close enough" matches. You embed both the new query and all cached queries into vector space, then find the nearest cached query by cosine similarity. If the similarity is above a threshold (e.g., 0.93), you return the cached response. "How do I reset my password?" and "How can I change my password?" would be very close in embedding space — cache hit.

When to use which:
- Exact-match: when inputs are generated by code (API integrations, template-based prompts) and will be literally identical
- Semantic: when inputs are typed by humans, who naturally rephrase things

</details>

**Q2: What is TTL (time-to-live) and why is it essential for caching?**

<details>
<summary>💡 Show Answer</summary>

**A:** TTL is the maximum age of a cached entry — after this time, the entry is automatically deleted and the next request triggers fresh inference. Without TTL, cached responses live forever and go stale.

Example of staleness causing harm: You cache the answer to "What is our return policy?" as "30 days". Six months later, the policy changes to "14 days". Without TTL, every user who asks that question gets the old (wrong) 30-day answer from cache.

Choosing TTL requires understanding how frequently your underlying information changes:
- Static content (mathematical facts, historical data): 7-30 days TTL
- Slow-changing content (company policies, product docs): 4-24 hours TTL
- Fast-changing content (pricing, inventory, news): 5-30 minutes TTL
- Real-time data: do not cache

A common mistake is setting no TTL at all, treating the cache as permanent storage. Always set one.

</details>

**Q3: What is prompt caching in the Anthropic and OpenAI APIs, and how does it save money?**

<details>
<summary>💡 Show Answer</summary>

**A:** Prompt caching is a provider-side feature where the provider caches the computed KV (key-value) attention state for a repeated prompt prefix. In transformer models, processing input tokens is expensive because each token generates a Key and Value matrix used in attention. For a 10,000-token system prompt, this is a lot of computation that repeats identically for every request.

With prompt caching enabled (by adding a `cache_control` marker on the cacheable part of your prompt), the provider processes those tokens once, stores the K/V state, and reuses it for subsequent requests with the same prefix. You are charged at approximately 10% of normal input token price for cached reads.

Example savings: A 10,000-token system prompt at $3/million tokens = $0.030 per request normally. With prompt caching at 90% cache hits and 10% price for cache reads: average cost = 10% × $0.030 + 90% × $0.003 = $0.0057 per request — an 81% reduction on the system prompt portion.

</details>

## Intermediate

**Q4: How would you choose a similarity threshold for semantic caching, and what happens when you set it too low vs too high?**

<details>
<summary>💡 Show Answer</summary>

**A:** The threshold (e.g., 0.93) is the minimum cosine similarity between the new query embedding and a cached query embedding required to return the cached response.

**Too low (e.g., 0.75):** Semantically different questions get treated as equivalent. "What is the return policy?" and "What are your business hours?" might both be about a store, and in a general embedding model might have similarity 0.78 — returning the wrong answer. False positive cache hits damage user trust.

**Too high (e.g., 0.99):** Almost nothing matches. The cache is barely better than exact-match. You pay for semantic search infrastructure but get few hits.

**How to choose the right threshold:**
1. Collect 100-200 pairs of real user queries
2. Have a human label: "same intent" or "different intent"
3. Compute cosine similarity for all pairs using your embedding model
4. Find the threshold that maximizes: (correct cache hits) / (total cache decisions) — essentially a precision/recall curve
5. Pick the threshold where precision (same-intent queries cached together) > 95%

Typical result: 0.92-0.95 works for most customer-facing applications. Adjust up if your domain has many closely-worded but meaningfully different questions (legal, medical).

</details>

**Q5: Describe the architecture of a production semantic cache. What components does it need?**

<details>
<summary>💡 Show Answer</summary>

**A:** A production semantic cache needs several components:

1. **Embedding model**: Converts query text to a dense vector. Options: `text-embedding-3-small` (OpenAI), `embed-english-v3` (Cohere), or a self-hosted model like `bge-small-en`. Important: the embedding model must stay consistent — if you change it, all existing cache entries become incompatible.

2. **Vector database / index**: Stores (embedding, cached_response, metadata) tuples and supports approximate nearest-neighbor (ANN) search. Options: Redis with vector search (simplest), Pinecone, Weaviate, or Qdrant for production scale.

3. **Cache key metadata**: Each entry should store: the original query text, embedding, response, model version, timestamp, TTL, and optionally: hit count (to identify frequently cached content for analysis).

4. **Similarity search layer**: Given a new query embedding, find the top-K nearest cached entries efficiently. ANN algorithms (HNSW, IVF) make this sub-millisecond even with millions of cached entries.

5. **Threshold filter**: Apply cosine similarity threshold after retrieval. Only return cached response if similarity ≥ threshold.

6. **Cache storage backend**: Redis is standard. For large-scale: Redis Cluster or a dedicated vector DB.

7. **Metrics collection**: Track per-request: hit or miss, similarity score on hits, cache age, latency for cache lookup vs inference. This is how you monitor threshold quality over time.

</details>

**Q6: How do you handle cache invalidation when your underlying data or model changes?**

<details>
<summary>💡 Show Answer</summary>

**A:** Cache invalidation is notoriously hard. Here are the patterns that work:

**Version-based invalidation**: Include a version identifier in the cache key.
```python
cache_key = f"llm:{model_version}:{prompt_hash}:{data_version}"
```
When you update the model or the underlying data, bump the version. Old cache entries are automatically missed (they have the old version in their key) and eventually expire via TTL.

**Event-driven invalidation**: When source data changes (e.g., product catalog update, policy change), publish an event that triggers cache deletion of all entries tagged with that data source.
```python
# On catalog update event:
redis.delete(*redis.keys("llm:*:catalog_*"))
```

**TTL as a fallback**: Even with event-driven invalidation, always set a TTL. Events can be missed; TTL ensures eventual consistency.

**Semantic cache challenge**: Unlike exact-match caches, semantic caches can't easily identify which cached entries are affected by a data change (because you don't know which natural-language question was about the changed data). Approach: either use very short TTLs for dynamic data, or tag each cache entry with the data sources it was derived from, and invalidate by tag.

</details>

## Advanced

**Q7: How would you implement a two-tier caching strategy for an LLM application serving millions of daily requests?**

<details>
<summary>💡 Show Answer</summary>

**A:** A two-tier cache separates fast/cheap exact hits from semantic hits:

**Tier 1 — In-memory exact cache** (< 1ms):
- Uses a local in-process dictionary or a local Redis instance
- Key: hash of the exact input (after normalization)
- Holds the last ~10,000 most frequent queries
- Hit rate typically 10-30% for FAQ-heavy workloads
- No network hop — sub-millisecond

**Tier 2 — Distributed semantic cache** (5-15ms):
- Redis Cluster with vector search, or Pinecone
- Stores embeddings + responses for all unique queries seen in last 30 days
- Similarity search returns hit if cosine > 0.93
- Hit rate typically 40-70% for conversational workloads

**Fallback — LLM inference** (200ms-5s):
- Only reached if both caches miss
- Result stored in both Tier 2 (embedding search) and Tier 1 (if becomes frequent)

```
Request
  → Tier 1 exact match check (< 1ms)
    → Hit: return immediately
    → Miss: Tier 2 semantic search (5-15ms)
      → Hit: return, promote to Tier 1 if high frequency
      → Miss: LLM inference → store in Tier 2
```

Expected performance at scale: 60-80% of requests served from cache, total P99 latency < 50ms for cache hits, ~$0 marginal cost per cache hit.

</details>

**Q8: What are the security and privacy implications of caching LLM responses in a multi-tenant system?**

<details>
<summary>💡 Show Answer</summary>

**A:** This is a critical concern in multi-tenant applications (SaaS, shared infrastructure):

**Problem 1 — Cross-user data leakage**: If User A asks about their account balance and you cache the response, User B asking the same question might receive User A's account information from cache. This is a catastrophic privacy violation.

**Solution**: Never cache responses that contain user-specific data. Scope cache keys to include user_id for any personalized content. Better: apply a "personal data classifier" before caching — if the response contains PII indicators (account numbers, names, private data), mark it as non-cacheable.

**Problem 2 — Prompt injection via cache poisoning**: An attacker might craft a query designed to look semantically similar to a common question, tricking the cache into returning attacker-controlled content to future users.

**Solution**: Only cache responses from your own model inference, never from external sources. Monitor for suspiciously high-hit entries on newly added cache entries. Apply output validation before caching.

**Problem 3 — Regulatory compliance (GDPR, CCPA)**: If cached data contains personal information and a user requests deletion ("right to be forgotten"), you must be able to delete all cached entries related to that user.

**Solution**: Tag every cache entry with the user_id(s) whose data contributed to it. Implement a cache invalidation API called during user deletion. Use TTLs that are short enough that data is purged within your compliance window.

**Best practice**: Only cache content that is:
- Not personalized to an individual user
- Safe to serve to any user in your system
- Explicitly reviewed for privacy implications

</details>
